Objective

A PoC of a hotswappable plugin based compiled service architecture to back SPA based development.

PoC Desired Qualities

  • Lower cost of development:
    • guardrailing into working in business-specific Logical Modules, rather than all over the plumbing too.
    • lower error/investigation/redo rate by using an Dependency Injection + ORM (EF) based development style.
    • reusable base covering system concerns, capable of handling multiple focused business cases on top of it.
  • Increase modularity:
    • Modularity by plugin architecture for Plugins(Calculators), Logical Capability Modules(People,Commerce,etc.), Environments (Azure/AWS).
    • While making plugins hot-swappable (a *very* hard task in .NET in general, compounded by being in a multi-user environment.)
    • While keeping security high (limiting access to specific core services).
    • While keeping development cost low (keeping service discoverability and injection).

PoC Discovery Tasks

  • [x] Usability: Lower the entry bar by using a recognisable ASP.Core standard template
  • [x] Replace the standard IServiceProvider with a more performance library (Lamar) capable of discovery by Reflection.
  • [x] Modularity: Register Controllers by Reflection irrespective of whether Referenced directly in solution, or in a well-known ("MODULES") directory.
  • [x] Discoverability: Ensure Controllers are OpenAPI discoverable.
  • [ ] Security: Provide transparent object mapping between internal entities and DTOs.
  • [ ] Maintainability: use of CodeFirst Object Definitions, by Reflection.
  • [ ] Intergratability: Ensure Controllers are OData capable.
  • [ ] Intergratability: Ensure Controllers are OpenAPI *and* OData capable.
  • [ ] Deployability: ensure Schemas can be developed from Powershell (Desktop or Pipeline) as well.
  • [ ] Deployability: ensure Schemas are separate yet can coexist (DDD at the EF managed Data level too).
  • [ ] Configurability: Ensure modules are droppable/replaceable (in-mem AppDomain, other).
  • [ ] Configurability: Portable *secure* configuration.
  • [ ] Caching: Weave it in as a default.
  • [ ] Usability: Only then work on the UX, consuming Apis.
  • [ ] Deployability: Develop Infrastructure as Code.
  • [ ] Deployability: Wrap with Pipeline as Code.
  • [ ] Productivity: Wrap with Development Environment as Code (Kanban).

Errata

Built with:

  • ASP.NET Core and C# for cross-platform server-side code
  • Angular and TypeScript for client-side code
  • Bootstrap for layout and styling

Using:

  • Client-side navigation. For example, click Counter then Back to return here.
  • Angular CLI integration. In development mode, there's no need to run ng serve. It runs in the background automatically, so your client-side resources are dynamically built on demand and the page refreshes when you modify any file.
  • Efficient production builds. In production mode, development-time features are disabled, and your dotnet publish configuration automatically invokes ng build to produce minified, ahead-of-time compiled JavaScript files.

The ClientApp subdirectory is a standard Angular CLI application. If you open a command prompt in that directory, you can run any ng command (e.g., ng test), or use npm to install extra packages into it.
